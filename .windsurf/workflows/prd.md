# PRD 文档生成工作流

## 使用方法
输入 `/prd` 命令启动 PRD 文档生成流程。

## 工作流步骤

### 1. 初始化 PRD 生成
- 确认用户要生成 PRD 的具体模块
- 收集基本的业务需求和背景信息
- 确认目标用户角色和使用场景

### 2. 需求信息收集
如果用户提供的信息不完整，按以下结构追问：

#### 2.1 基础信息收集
- **需求背景**：为什么要做这个功能？解决了什么业务痛点？
- **用户故事**：从用户角度描述使用场景和期望
- **业务流程**：完整的操作流程是怎样的？涉及哪些角色？
- **功能范围**：包含哪些具体功能点？排除哪些功能？
- **数据结构**：涉及哪些核心数据实体？它们之间的关系？
- **约束条件**：有哪些业务规则、权限限制、技术约束？
- **异常场景**：可能出现的异常情况有哪些？如何处理？

#### 2.2 标准化需求梳理提示词
参考配置管理交付件配置文档格式，按以下结构梳理需求：

```
请按以下格式详细描述您的需求：

## 功能描述
[简要描述该功能的核心价值和主要作用]

## 优先级
[P0/P1/P2/P3 - 说明优先级和原因]

## 输入条件/前置条件
[功能启动的必要条件，包括数据来源、权限要求等]

## 处理流程

### 1. 界面布局与交互
- 页面主要区域划分
- 关键控件和操作元素
- 用户交互流程

### 2. 数据列表区域
- 列表显示字段及类型
- 字段说明和显示规则
- 默认值和数据来源
- 排序和筛选条件

### 3. 操作功能
- **新增**：[描述新增操作的流程和校验]
- **编辑**：[描述编辑操作的流程和限制]
- **删除**：[描述删除操作的确认机制和影响]
- **查询**：[描述查询条件和结果显示]
- **其他操作**：[批量操作、导入导出等]

### 4. 业务逻辑
- 核心业务规则
- 数据校验规则
- 权限控制逻辑
- 状态流转规则

## 输出结果/后置操作
[功能完成后产生的结果和后续操作]

## 异常处理
- [可能的异常情况1]：[处理方式]
- [可能的异常情况2]：[处理方式]

## 性能要求
- 响应时间要求
- 并发处理能力
- 数据量处理能力

## 集成要求
- 需要对接的其他系统
- 数据同步要求
- 接口规范要求
```

#### 2.3 需求完整性检查清单
在生成PRD前，确认以下信息已收集完整：
- ✅ 功能边界清晰（做什么，不做什么）
- ✅ 用户角色和权限明确
- ✅ 业务流程完整（正常流程+异常流程）
- ✅ 数据结构详细（字段定义、类型、约束）
- ✅ 界面交互规则（页面布局、操作流程）
- ✅ 技术约束明确（性能、安全、集成）
- ✅ 验收标准可量化

### 3. 生成 PRD 文档
按照标准模板生成 PRD 文档：

#### 3.1 文档概述
- 项目背景（专业重述）
- 用户价值分析
- 适用范围定义

#### 3.2 业务流程图
- 使用 Mermaid 绘制泳道图
- 区分主流程和异常流程
- 明确各角色职责

#### 3.3 详细功能需求
- 前置条件
- 页面元素与交互
- 表单校验规则
- 核心逻辑规则
- 后置操作

#### 3.4 数据字典
- 字段名称、类型、是否必填
- 默认值、校验规则
- 数据来源说明

#### 3.5 异常流程与边界情况
- 网络异常处理
- 数据并发冲突
- 业务异常场景
- 系统边界情况

#### 3.6 埋点与统计需求
- 用户行为统计
- 业务指标监控
- 系统性能指标

### 4. 质量检查
- 检查逻辑完整性
- 验证业务合理性
- 确认开发可行性
- 补充遗漏的异常场景

### 5. 文档输出
- 保存到 `_draft/PRD/{需求名}` 目录
- 按需求名和日期命名文件
- 生成可交付的完整 PRD 文档

## 注意事项

### 信息完整性要求
只有当以下信息都明确时，才生成完整 PRD：
- 完整的业务流程
- 明确的角色权限
- 详细的数据结构
- 全面的异常处理

### 逻辑补全原则
基于行业标准自动补全：
- 软件交付系统标准流程
- 数据安全规范
- 用户体验标准

### 文档质量标准
- 逻辑严密，无歧义
- 开发人员可直接理解
- 测试人员可编写用例
- 产品人员可验收功能

## 输出示例

生成的 PRD 文档将包含：
- 清晰的功能边界
- 完整的业务流程
- 详细的技术规格
- 全面的异常处理
- 可执行的验收标准
